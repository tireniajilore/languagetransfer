## Problem Statement

VoiceAI is an interactive Spanish tutor built on the Language Transfer "Thinking Method." It has 3 working lessons with ElevenLabs TTS (segmented Spanish/English pacing), a lesson engine state machine, and a pause-respond-continue interaction loop. The builder wants to know: does anyone actually want this? Before polishing or scaling to 100+ lessons, the priority is getting the current scrappy version in front of real language learners and measuring whether they engage.

## What Makes This Cool

Language Transfer's Thinking Method forces you to construct the answer before hearing it. The deliberate pause where you think "how would I say that?" is where the actual learning happens. Every AI language app in 2026 is racing toward free-form conversation (Langua, Speak, Praktika, Talkio). None of them use this pedagogy. VoiceAI is the only interactive implementation of the Thinking Method — turning passive audio lessons into an app that pauses, waits for you to think and speak, then continues.

## Constraints

- Solo builder, learning/having fun — but with a real instinct to validate before over-investing
- 3 lessons currently working (Lessons 2, 3, and 4 of Language Transfer Complete Spanish — the app starts at Lesson 2, matching the LT course numbering where Lesson 1 is introductory)
- ElevenLabs TTS with segmented Spanish/English audio and pacing controls
- Next.js frontend, no backend integration yet (lessons loaded client-side)
- No AI evaluation of spoken responses (engine records but doesn't evaluate)
- Web app only — no mobile native

## Premises

1. The LT Thinking Method is effective pedagogy, and interactive pause-and-respond improves it over passive audio.
2. The current scrappy version is good enough to test demand — polish comes after validation.
3. Target user is any Spanish learner, not just LT fans.
4. ElevenLabs TTS quality is sufficient for a test.
5. Web app in a browser is sufficient — no native mobile app needed for validation.

## Approaches Considered

### Approach A: Polish the Flow
Focus entirely on UX polish of existing 3 lessons. Smooth transitions, loading states, progress persistence, visual design. No new features.
- Effort: S (CC: ~1 hour)
- Risk: Low
- Rejected because: builder wants to validate demand before investing in polish.

### Approach B: Polish + Pipeline
Polish UX AND build lesson ingestion pipeline for scaling to 100+ lessons. Includes progress tracking and lesson selection.
- Effort: M (CC: ~2-3 hours)
- Risk: Med
- Rejected because: premature scaling before demand signal.

### Approach C: Ship and Learn (SELECTED)
Deploy the current scrappy version. Add minimal instrumentation to know if people engage. Get it in front of 10 language learners this week. Measure completion rates and return visits.
- Effort: S (CC: ~30 min)
- Risk: Low
- The builder's own instinct — don't polish what nobody wants.

## Recommended Approach: Ship and Learn

### Current user flow (what testers will experience)
1. User opens the URL in a browser
2. Lesson 2 loads automatically (Latin Cognates — AL Endings)
3. ElevenLabs voice speaks an English explanation or prompt
4. When the tutor asks "how would you say X in Spanish?", the app pauses and waits
5. User thinks, then speaks their answer aloud (or types it)
6. The app reveals the correct answer and continues to the next step
7. After all steps, the lesson ends (engine reaches `completed` mode)
8. No lesson selector yet — only Lesson 2 loads from `lesson-player.tsx`

### What to deploy
The current version with minimal changes. The only new code is lightweight analytics instrumentation and a lesson selector to expose all 3 lessons. No user-facing polish. The default input mode is text entry (browser speech recognition adapters exist but are not the primary path for this test).

### Deployment options
1. **Vercel** (recommended) — `npx vercel` from the `lt-runner` directory. Free tier. Instant deploy. Shareable `.vercel.app` URL with HTTPS. Zero config for Next.js. The default subdomain is sufficient for a demand test — no custom domain needed.
2. **Netlify** — similar, slightly more setup.
3. **GitHub Pages** — requires static export, more friction.

### Minimal instrumentation
Add lightweight analytics to answer three questions:
1. **Do people start?** — Track "lesson started" events
2. **Do people finish?** — Track "lesson completed" events (per lesson). A lesson is "completed" when the engine reaches the `completed` mode (terminal state of the state machine — all steps played).
3. **Do people come back?** — Track returning visitors (localStorage flag + session count). Note: localStorage won't survive incognito/cross-device usage, but at n=10 testers, qualitative follow-up ("did you come back?") is the real signal anyway.

**Recommended: Simple localStorage** — sufficient at n=10 testers. Write a session counter and per-lesson completion flags to localStorage. No third-party analytics needed for this scale. If the test works and you want to scale distribution, upgrade to Vercel Analytics (free tier, zero config) later.

### How to get 10 learners this week
1. Text/DM 5 friends who've mentioned wanting to learn Spanish
2. Post in a language learning subreddit (r/languagelearning, r/learnspanish) — "I built an interactive version of Language Transfer, looking for feedback"
3. Post in the Language Transfer community (they have a forum and Discord)
4. Share on Twitter/X with a short demo video (screen recording of a lesson)

### What to measure
- **Completion rate:** What % finish all 3 lessons within the one-week test period?
- **Drop-off point:** Where in lesson 2 do people stop? (The first lesson is the critical one)
- **Qualitative feedback:** Ask each tester: "Would you do lesson 4 if it existed?"
- **Return rate:** Does anyone come back a second day?

### Success criteria
- If 3+ out of 10 testers finish all 3 lessons (within the week) → strong signal, worth building more
- If 5+ out of 10 say "yes I'd do lesson 4" → build the pipeline (Approach B)
- If <2 finish lesson 2 → the interaction model needs rethinking before scaling

## ElevenLabs Cost Protection

The TTS API route (`/api/tts`) is server-side on Vercel, so the API key is not exposed to the browser. However, each lesson step triggers a TTS request, and public sharing (Reddit, Discord) could produce unexpected traffic spikes.

**Required before deploy:**
- Set a hard monthly spending cap on the ElevenLabs account dashboard

**Recommended (defer if needed):**
- For the n=10 test, the ElevenLabs dashboard spending cap is sufficient. If you later share more broadly (Reddit post gets traction), add a server-side request counter via Vercel KV (free tier, ~15 min setup) that returns a browser TTS fallback after 500 requests/day. Note: in-memory counters will not work on Vercel serverless since each invocation is a fresh function instance.
